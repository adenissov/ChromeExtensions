‚ĚĮ The issue I have with this extension is that the position of the line it adds 
   to the right-click context menu depends on alphabetical sorting. can this    
  position be changed according to other rules in the web page in DOM after the 
   page is opened??                                                             

‚óŹ Short answer: no ‚ÄĒ Chrome's chrome.contextMenus API doesn't expose positional
  control. You can't move the item relative to native entries (Copy/Paste/etc.)
  or reorder it based on DOM state at runtime. Chrome decides the layout:
  extension items are grouped together and, when multiple extensions contribute,
   sorted alphabetically by title.

  What you can do:

  1. Title-prefix trick (cheap, works today). Alphabetical sort is by the title
  string, so a leading space, !, #, or a digit pushes your item to the top of
  the extensions group. Example: change title: 'SR Search Integration Requests'
  to title: ' SR Search Integration Requests' or title: '‚Ď† SR Search Integration
   Requests'. It's hacky but it's how most extensions win the ordering battle.
  2. Dynamic title via chrome.contextMenus.update(). You can change the title
  (and therefore its sort position) from background.js in response to messages
  from the content script. So if the DOM says "this page has feature X," you
  could swap the title to a prefixed form. But you still can't pick an absolute
  slot ‚ÄĒ you're only nudging alphabetical order.
  3. Custom HTML menu (full control, but intrusive). Listen for contextmenu in
  content.js, call event.preventDefault(), and render your own floating div. You
   get exact positioning and DOM-aware logic, but you lose the native
  Copy/Paste/Inspect items unless you rebuild them, and users generally dislike
  this.

  My recommendation: if you just want it higher in the list, try option 1 first
  ‚ÄĒ it's a one-character change in manifest.json/background.js. Want me to apply
   it?

‚ĚĮ does character case matter during that sorting? What goes higher: "i" or "I"? 

‚óŹ Chrome's context menu uses case-insensitive locale-aware sorting, so "i" and
  "I" are treated as equal ‚ÄĒ ties fall back to insertion order (which extension
  registered first).

  Practically: don't rely on case to control ordering. If you want a guaranteed
  top slot, use a prefix that sorts before letters regardless of case ‚ÄĒ a space,
   !, #, *, or a digit. Those beat any A‚ÄďZ / a‚Äďz title.

  If you want certainty for your specific Chrome/Edge version, install two test
  menu items with titles "a test" and "A test" and see which wins ‚ÄĒ takes 30
  seconds.
